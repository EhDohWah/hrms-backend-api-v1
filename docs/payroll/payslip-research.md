## 1. System Overview

The payslip PDF is generated on demand from an already-saved `Payroll` database record. The system does **not** calculate payroll at the time of PDF generation — all numbers are pre-computed and stored when the payroll record is created. The PDF layer is purely a presentation concern.

There are two paths for getting data into the `payrolls` table:

| Path | Description | Triggers |
|---|---|---|
| **Bulk auto-calculation** | `PayrollService` computes all 13 payroll items from scratch using employment data, benefit settings, and YTD history | `POST /api/v1/payroll/bulk` → queue job |
| **Manual import** | User uploads an Excel file with pre-computed values; no calculations performed | `POST /api/v1/payroll/upload` → queue job |

Once a `Payroll` record exists, the payslip PDF is generated by:

```
GET /api/v1/payrolls/{payroll}/payslip
```

The record is looked up, relationships are eager-loaded, a `$data` array is built, and `dompdf` renders the correct Blade template to a PDF stream.

## 2. HTTP Request → PDF Response Flow

### 2.1 Middleware Stack

The request passes through global middleware in this order before reaching the controller:

1. **`SecurityHeaders`** — appends security response headers (HSTS, X-Frame-Options, etc.)
2. **`ForceJsonResponse`** — forces `Accept: application/json` on all API requests (note: PDF response bypasses this since the service returns a `StreamedResponse`, not JSON)
3. **`AuthenticateFromCookie`** — reads the `auth_token` HttpOnly cookie and injects it as a `Bearer` token, enabling Sanctum authentication
4. **`ThrottleRequests:api`** — rate limiting
5. **`auth:sanctum`** — authenticates the user from the Bearer token
6. **`permission:employee_salary.read`** — Spatie Permission gate on the route; 403 if not authorized

### 2.2 Route

**File:** `routes/api/payroll.php` (line 20)

```php
Route::get('/{payroll}/payslip', [PayrollController::class, 'generatePayslip'])
    ->middleware('permission:employee_salary.read');
```

The `{payroll}` segment triggers Laravel's **implicit route model binding** — the `Payroll` model is resolved by primary key before the controller runs. Soft-deleted payrolls return a **404** (model binding uses `findOrFail()` which excludes soft-deleted records by default).

### 2.3 Controller

**File:** `app/Http/Controllers/Api/V1/PayrollController.php` (line 186)

```php
public function generatePayslip(Payroll $payroll)
{
    return $this->payrollService->generatePayslip($payroll);
}
```

The controller is a pure pass-through. There is **no Form Request** — access control is entirely handled by the `permission:employee_salary.read` middleware. The service return value (a `StreamedResponse`) is returned directly to the browser.

### 2.4 Service: Data Loading

**File:** `app/Services/PayrollService.php` (line 320)

The service begins with a single eager-load call to avoid N+1 queries:

```php
$payroll->load([
    'employment.employee',
    'employment.department:id,name',
    'employment.position:id,title',
    'employment.site:id,name',
    'employeeFundingAllocation.grantItem.grant',
    'grantAllocations',                          // PayrollGrantAllocation snapshots
]);
```

This triggers up to 6 SQL queries (one per relationship chain, deduplicated by eager loading).

### 2.5 Service: Grant Data Resolution (Two-Tier Fallback)

The system resolves grant metadata from two sources in priority order:

```php
$grantAllocation  = $payroll->grantAllocations->first();   // 1st: historical snapshot
$fundingAllocation = $payroll->employeeFundingAllocation;  // 2nd: live funding record

$grantCode      = $grantAllocation?->grant_code        ?? $fundingAllocation?->grantItem?->grant?->code         ?? 'N/A';
$grantName      = $grantAllocation?->grant_name        ?? $fundingAllocation?->grantItem?->grant?->name         ?? 'N/A';
$budgetLineCode = $grantAllocation?->budget_line_code  ?? $fundingAllocation?->grantItem?->budgetline_code      ?? 'N/A';
$grantPosition  = $grantAllocation?->grant_position    ?? $fundingAllocation?->grantItem?->grant_position       ?? 'N/A';
$fte            = $grantAllocation?->fte               ?? $fundingAllocation?->fte                              ?? 0;
```

**Why two sources?**

- `PayrollGrantAllocation` is a **snapshot** model — it captures grant metadata at the time the payroll was created. If a grant is renamed or reassigned later, the payslip still shows the original data. This is the preferred source.
- `EmployeeFundingAllocation → GrantItem → Grant` is the **live chain** — used as a fallback for older payroll records created before the snapshot system existed, or for manually imported records.
- Final fallback: the string `'N/A'`.

### 2.6 Service: Building the View Data Array

```php
$data = [
    'payroll'        => $payroll,                                         // Full Payroll model (encrypted fields auto-decrypted)
    'employee'       => $employee,                                        // Employee model
    'employment'     => $employment,                                      // Employment model
    'department'     => $employment?->department?->name  ?? 'N/A',       // String (not object)
    'position'       => $employment?->position?->title   ?? 'N/A',       // String (not object)
    'site'           => $employment?->site?->name         ?? 'N/A',       // String (not object)
    'grantCode'      => $grantCode,                                       // Resolved above
    'grantName'      => $grantName,
    'budgetLineCode' => $budgetLineCode,
    'grantPosition'  => $grantPosition,
    'fte'            => $fte,                                             // Raw decimal e.g. 0.75
    'ftePercentage'  => round((float) $fte * 100, 2),                    // e.g. 75.0
    'payPeriod'      => Carbon::parse($payroll->pay_period_date)->format('F Y'), // e.g. "January 2025"
];
```

### 2.7 Service: Template Selection, PDF Generation, and Streaming

```php
// Organisation-based template routing
$view = $employee?->organization === 'BHF' ? 'pdf.bhf-payslip' : 'pdf.smru-payslip';

$pdf = Pdf::loadView($view, $data);
$pdf->setPaper('a5', 'landscape');

$staffId  = $employee?->staff_id ?? 'unknown';
$period   = Carbon::parse($payroll->pay_period_date)->format('Y-m');
$filename = "payslip-{$staffId}-{$grantCode}-{$period}.pdf";

return $pdf->stream($filename);  // StreamedResponse: Content-Type: application/pdf, inline disposition
```

- Uses `barryvdh/laravel-dompdf` via the `Pdf` facade
- Paper size is hardcoded: **A5 landscape**
- Filename pattern: `payslip-{staff_id}-{grant_code}-{YYYY-MM}.pdf`
- `stream()` returns `Content-Disposition: inline` — the browser renders PDF inline, not as a download

## 4. Payroll Database Schema — All Fields

**File:** `database/migrations/2025_04_27_114136_create_payrolls_table.php`

Every monetary column is stored as SQL `text` because Laravel's `'encrypted'` cast serializes and encrypts the value before writing to the database. This has a critical side-effect: **monetary fields cannot be aggregated at the SQL level** (`SUM()`, `AVG()`, `ORDER BY` on amounts are impossible). All aggregations happen in PHP by loading records into memory.

| Column | Type | Nullable | Purpose |
|---|---|---|---|
| `id` | bigint (auto) | No | Primary key |
| `employment_id` | FK → employments | No | Links to the employment record |
| `employee_funding_allocation_id` | FK → employee_funding_allocations | No | Links to the specific grant allocation that funded this payroll |
| `gross_salary` | text (encrypted) | No | Full monthly contractual salary — before pro-rating or FTE split |
| `gross_salary_by_FTE` | text (encrypted) | No | Pro-rated salary × FTE — the actual amount charged to the grant |
| `retroactive_adjustment` | text (encrypted) | Yes | Deferred salary from prior month (positive = underpaid, negative = overpaid) |
| `thirteen_month_salary` | text (encrypted) | No | 13th month payout — non-zero only in December |
| `thirteen_month_salary_accured` | text (encrypted) | No | YTD running projection of December payout — for HR tracking only, not shown on payslip |
| `pvd` | text (encrypted) | Yes | Employee Provident Fund deduction (Thai Local ID only, 7.5%) |
| `saving_fund` | text (encrypted) | Yes | Employee Saving Fund deduction (Non-Thai Local Non-ID only, 7.5%) |
| `employer_social_security` | text (encrypted) | No | Employer SSF contribution (5%, capped ฿875, FTE-proportional) |
| `employee_social_security` | text (encrypted) | No | Employee SSF deduction (same rates and cap as employer) |
| `employer_health_welfare` | text (encrypted) | No | Employer HW contribution (Non-Thai eligible orgs, tiered by salary) |

| `total_salary` | text (encrypted) | No | Total cost to company = `gross_by_FTE + retro + 13th + bonus + employer_ss + employer_hw` |
| `total_pvd` | text (encrypted) | No | Combined PVD: employee + employer match |
| `total_saving_fund` | text (encrypted) | No | Combined Saving Fund: employee + employer match |

| `employer_contribution` | text (encrypted) | No | Total employer cost = `employer_ss + employer_hw + pvd_employer_portion` |
| `total_deduction` | text (encrypted) | No | Total employee deductions = `pvd + saving_fund + employee_ss + employee_hw + tax` |

| `pay_period_date` | date | No | The pay period date — used for financial period grouping |
| `created_at` / `updated_at` | timestamps | Yes | Standard Eloquent timestamps |

## 5. How Each Payroll Field Gets Its Value (Calculation Engine)

The central method is `PayrollService::calculateAllocationPayroll()` (`PayrollService.php` line 848). It executes 13 steps in sequence, each in a dedicated private method.

### Step 1 — `gross_salary` (line 1218)

```php
$employment->getSalaryAmountForDate($payPeriodDate)
```

Returns the contractual monthly salary from the employment record. Uses a **date-aware lookup** so retroactive payroll recalculations use the correct historical salary (not the current one, which may have changed).

### Step 2 — `gross_salary_by_FTE` (line 1233)

Implements the **30-day standardized month rule**:

| Start Day | Days Worked | Result |
|---|---|---|
| 1st | 30 | Full month pay |
| 2nd–15th | `30 - startDay + 1` | Partial pay (inclusive of start day) |
| 16th+ | 0 (deferred) | No pay this month; deferred to next month as retroactive |

Resignation mid-month: `daysWorked = min(resignDay, 30)`.

Formula: `round((adjustedGrossSalary / 30) * daysWorked * fte)`

> **Note:** Pro-rating is applied to the full salary first, then FTE is applied. This avoids rounding drift when the same employee has multiple allocations (e.g., 80/20 split — rounding the daily rate separately per allocation would accumulate errors).

### Step 3 — `retroactive_adjustment` (line 1284)

Only triggered when the employee started on **day >= 16 in the previous month** (meaning they received zero pay that month, and the deferred days are now paid).

Deferred days: `30 - startDay + 1` (e.g., start on 17th → 14 deferred days).

Formula: `round((grossSalary / 30) * deferredDays * fte)`

### Step 4 — `thirteen_month_salary` (line 1328)

Non-zero **only in December**. Eligibility starts from `employment.start_date` — no probation gate.

Algorithm:
1. Query all prior `gross_salary_by_FTE` records for **this specific allocation** in the current year from the `payrolls` table (YTD, excluding current month since it's not yet saved)
2. Add current month's `grossSalaryCurrentYearByFTE`
3. Divide by the divisor (default: 12, configurable via `PayrollPolicySetting.thirteenth_month_divisor`)

Formula: `round((ytdGrossByFTE + currentMonthGrossByFTE) / divisor)`

> **Important:** The 13th month is per-allocation. If an employee switches grants mid-year, each grant pays its own proportional share based on what it actually funded.

> **Also note:** The `thirteen_month_salary_accured` field is a monthly running projection stored for HR tracking, but it is **never displayed on the payslip**.

### Step 5 — `pvd` and `saving_fund` (line 1367)

Only deducted after passing probation (uses end-of-month comparison so transition month is included). Mutually exclusive by nationality:

| Employee Status | Toggle Required | Field Used | Rate Source |
|---|---|---|---|
| Local ID (Thai) | `employment.pvd = true` | `pvd` | `BenefitSetting::KEY_PVD_EMPLOYEE_RATE` (default 7.5%) |
| Local Non-ID (Non-Thai) | `employment.saving_fund = true` | `saving_fund` | `BenefitSetting::KEY_SAVING_FUND_EMPLOYEE_RATE` (default 7.5%) |
| Expats | N/A | Both 0 | — |

### Step 6 & 7 — `employer_social_security` / `employee_social_security` (lines 1416, 1445)

Applies to **Local ID and Local Non-ID only** (Expats are excluded).

Rules (sourced from `BenefitSetting`):
- Rate: 5% of effective salary
- Minimum salary for SSF: ฿1,650
- Maximum salary for SSF calculation: ฿17,500
- Monthly cap per person: ฿875

Formula: `round(min(effectiveSalary * 0.05, 875) * fte)`

Calculated on the **full salary** first, then multiplied by FTE to avoid cap-splitting issues across multiple allocations.

### Step 8 & 9 — `employer_health_welfare` / `employee_health_welfare` (lines 1472, 1517)

Three-tier system based on **full monthly salary** thresholds (configurable via `BenefitSetting`):

| Salary Range | Non-Thai Employee | Thai Employee | Non-Thai Employer |
|---|---|---|---|
| ≤ ฿5,000 | ฿30 | ฿50 | ฿60 |
| ฿5,001–15,000 | ฿50 | ฿80 | ฿100 |
| > ฿15,000 | ฿75 | ฿100 | ฿150 |

- **Expats:** `employee_health_welfare = 0` (skipped entirely)
- **Employer HW:** Only paid for Non-Thai employees at eligible organizations (per `BenefitSetting::getEmployerHWEligibility()`)
- All amounts are multiplied by FTE

### Step 10 — `tax` (line 1568) — ACM Method

Tax is a **personal obligation** calculated on the full gross salary (not the FTE portion). The FTE-adjusted amount is only used at the very end to split the tax cost across allocations.

Process:
1. Query all prior payrolls for this **employment** (all allocations) in the current year to aggregate: `ytd_income`, `ytd_tax_withheld`, `ytd_ssf`, `ytd_pvd_sf`
2. Scale the current month's FTE-level SSF and PVD back to full-salary level (divide by FTE)
3. Call `TaxCalculationService::calculateEmployeeTax()` with:
   - Full gross salary
   - YTD aggregates
   - Employee deductions (spouse, children, eligible parents)
4. Tax service computes: `estimated_annual_income = ytd_income + (currentSalary × remaining_months_this_year)`; applies progressive brackets; derives monthly withholding: `(estimated_annual_tax - ytd_withheld) / remaining_months`
5. Final: `round(fullMonthlyTax * fte)` — splits the full-salary tax across allocations proportionally

### Step 11 — `net_salary` (line 1635)

```
net_salary = round(total_income - total_deduction)
```

### Step 12 — `total_salary` (line 1655)

Total cost to company:

```
total_salary = round(gross_by_FTE + retroactive + thirteen_month + salary_increase + employer_ss + employer_hw)
```

### Step 13 — `total_pvd` / `total_saving_fund` (line 1677)

Combined employee + employer PVD:

```
total_pvd = round(pvd_employee + pvd_employee × (employerRate / employeeRate))
```

Since both default to 7.5%, this typically equals `pvd × 2`.

### Derived Aggregate Fields

| Field | Formula |
|---|---|
| `total_income` | `gross_by_FTE + retroactive_adjustment + thirteen_month_salary + salary_increase` |
| `total_deduction` | `pvd + saving_fund + employee_social_security + employee_health_welfare + tax` |
| `employer_contribution` | `employer_social_security + employer_health_welfare + pvd_employer_portion` |
| `thirteen_month_salary_accured` | `round((ytdGrossByFTE + currentMonthGrossByFTE) / divisor)` — monthly running accrual |

## 7. Bulk Payroll Flow (Auto-Calculation)

### 7.1 Trigger

```
POST /api/v1/payroll/bulk
Body: { pay_period_date, filters: { subsidiaries: ["SMRU", "BHF"] } }
```

### 7.2 `BulkPayrollService::createBatch()` (synchronous)

1. Queries active `Employment` records matching the filters (also includes employees who resigned mid-month, to ensure their final payroll is processed)
2. Creates a `BulkPayrollBatch` record with `status = 'pending'`
3. Dispatches `ProcessBulkPayroll::dispatch($batchId, $payPeriodDate, $employmentIds)` to the queue
4. Returns `{ batch_id, pay_period_date, total_employees, status }` immediately — the client doesn't wait for processing

### 7.3 `ProcessBulkPayroll::handle()` (queue job)

```
batch.status → 'processing'

Load all employments with:
  employee (+ children, for tax deductions)
  department, position
  active EmployeeFundingAllocations grouped by employee_id

December only: also load historical (inactive/closed) allocations
  from YTD payroll records for the 13th-month-only calculation

For each employment:
  For each active funding allocation:
    1. PayrollService::calculateAllocationPayrollForController()
         → Returns calculations[] with all 13 payroll items
    2. ProcessBulkPayroll::preparePayrollRecord()
         → Maps calculation keys → DB column names (see §11)
    3. Push to $payrollBuffer[]

    Every 10 records:
      insertPayrollBatch()  ← uses Payroll::create() in DB::transaction
        → Also creates PayrollGrantAllocation snapshot per record
        → Calls createInterOrganizationAdvanceIfNeeded()
           (creates InterOrganizationAdvance if grant.org ≠ employee.org)
      broadcastProgress()   ← WebSocket event

December historical allocations (inactive grants that had payrolls this year):
  calculateHistoricalAllocation13thMonth()
    → Returns same structure but all salary fields = 0, only 13th month non-zero

Insert remaining buffer.
batch.status → 'completed'
Final WebSocket broadcast.
```

### 7.4 WebSocket Progress Events

**Event:** `app/Events/PayrollBulkProgress.php`
**Channel:** `payroll-bulk.{batchId}` (public channel)
**Event name:** `payroll.progress`

```json
{
  "batchId": 42,
  "processed": 30,
  "total": 85,
  "status": "processing",
  "currentEmployee": "John Smith",
  "currentAllocation": "GRANT-001 - Research Officer (100%)",
  "stats": {
    "successful": 28,
    "failed": 2,
    "advances_created": 5
  }
}
```

Broadcast every `BROADCAST_EVERY_N_PAYROLLS = 10` records. Not broadcast at the last item to prevent race-condition UI flash. Completion triggers a final dedicated broadcast.

Clients can also poll `GET /api/v1/payroll/bulk/{batch}/status` as a WebSocket fallback.

## 10. Bugs & Issues Found

### 10.1 Employer PVD Displayed Using the Wrong Field

In the "Other" column (employer contributions), row 2 shows "Provident fund SMRU/BHF 7.5%" and uses `$payroll->pvd` — which is the **employee's** PVD deduction, not the employer's matching contribution. The employer PVD amount would be `total_pvd - pvd` (or the `pvd_employer` key in the calculation result), but this is not stored as a separate database column. The result: both the employee and employer PVD rows show the same identical number.

**Affected lines:**
- `smru-payslip.blade.php` line 175: `{{ number_format((float) $payroll->pvd, 2) }}`
- `bhf-payslip.blade.php` line 175: same

**Fix options:**
1. Add a `pvd_employer` column to the `payrolls` table and store it during calculation
2. Or compute it in the template: `$payroll->total_pvd - $payroll->pvd`

### 10.2 Employer Contributions Duplicated in Deductions Column

Rows 5, 6, and 7 of the deductions column show employer PVD, employer SSF, and employer HW again (same values as the "Other" column). This is a visual design choice — showing the company's burden alongside employee deductions — but the duplication means the `total_deduction` column subtotal does not match the sum of the visible items in the deductions column (since `total_deduction` in the database contains only employee-side deductions).

### 10.3 `saving_fund` Never Appears on the Payslip

Non-Thai (Local Non-ID) employees have their contribution stored in `saving_fund` instead of `pvd`. The payslip always reads `$payroll->pvd` for the employee provident fund row. For Non-Thai employees, `pvd` will be `0` and the actual deduction is in `saving_fund`, which is never displayed. **Non-Thai employee payslips show incorrect (zero) contribution in the Provident Fund row.**

### 10.4 Date Shows Server's Current Date, Not Pay Period

The payslip template uses `now()->format('d/m/Y')` for the "Date:" line. The variable `$payPeriod` (containing the pay period formatted as "January 2025") is passed in the `$data` array but never used. A payslip printed months after the pay period will show the printing date, not the period it covers.

**Fix:** Replace `{{ now()->format('d/m/Y') }}` with `{{ Carbon::parse($payroll->pay_period_date)->format('d/m/Y') }}` — or use the passed `$payPeriod` variable (though that's a different format).

### 10.5 Unused Variables in `$data` Array

These variables are built and passed to the template but never rendered:

| Variable | Built As | Where It Could Be Used |
|---|---|---|
| `$employment` | Employment model | Department, site info |
| `$department` | String | Could be shown in the header |
| `$site` | String | Could be shown in the header |
| `$grantPosition` | String | Could appear as job title from grant |
| `$fte` | Float | Redundant — `$ftePercentage` is used |
| `$payPeriod` | "January 2025" | Replace `now()` with this for the date row |

### 10.6 No Employer PVD Database Column

The calculation engine computes `pvd_employer` (lines 1096–1097 in `PayrollService.php`) and `total_pvd` (combined), but there is no standalone `employer_pvd` column in the database. The payslip cannot accurately display the employer's separate PVD contribution without either adding a column or computing it in the template.

### 10.7 `organization` Comparison is Not Enum-Safe

The template routing uses a raw string comparison (`=== 'BHF'`). If `employee.organization` is backed by a PHP enum, the comparison `EnumInstance === 'BHF'` always returns `false` and all employees silently get the SMRU template. Currently `organization` appears to be a plain string field (no `Organization` enum class exists in `app/Enums/`), so this is safe today — but needs attention if an enum is introduced later.

### 10.8 Import Bypasses Encryption (Data Integrity)

`PayrollsImport.php` uses `Payroll::insert()` which bypasses the `encrypted` cast. Imported records are stored as plaintext. Calculated records are stored encrypted. Mixing both in the same table means some records are readable as plaintext from the database while others are not — inconsistent security posture and potential issues for any code that reads the DB directly.

### 10.9 Thirteen Month Salary Accrual Typo

The column and code use `thirteen_month_salary_accured` (missing the second 'c'). This is in the migration, model, and calculation code. It's a cosmetic issue but would require a migration to fix.
